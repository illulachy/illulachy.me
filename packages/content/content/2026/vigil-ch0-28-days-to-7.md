I need to tell you about a number: 28.

That's how many days a pharmaceutical company spends, on average, reviewing the clinical documents for a single trial endpoint. Not 28 days of active work. 28 days of calendar time — reviewers scheduling time, reading documents, reconciling disagreements, writing findings, going back to the documents, repeating.

The documents themselves are dense. A single Clinical Study Report can be 2,000 pages. The Statistical Analysis Plan that governs it is another 300. The tables inside are generated by SAS macros against data nobody outside the trial team has ever seen. Reviewers are trained to look for specific things — protocol deviations, missing patient populations, p-value manipulation through endpoint selection — but "trained to look for" is not the same as "can see in a 2,000-page document in one sitting."

The question we started with was simple: what if the model read the documents first?

Not to replace the reviewer. To prepare the ground. To surface the things that match known risk patterns before a human opens the document. To reduce 28 days of calendar time to 7 — not because we found a shortcut, but because we removed the waiting.

This is the story of building Vigil.

---

## What We Were Actually Building

Vigil is an agentic clinical document reviewer. Enterprise SaaS, built for pharma companies and CROs (contract research organizations) who need to review clinical trial documentation as part of regulatory submission preparation.

The regulatory context matters. 21 CFR Part 11 is the FDA regulation that governs electronic records in clinical trials. It requires: audit trails for every action, electronic signatures that are legally equivalent to handwritten ones, access controls that prevent unauthorized modification, and validation documentation that proves the software does what it claims.

We weren't building a chatbot. We were building a regulated system. Every design decision had to survive an FDA audit.

The stack: FastAPI (Python) for the backend, Azure OpenAI for the AI layer, Azure Document Intelligence for OCR and layout extraction, PostgreSQL for structured data, Redis for caching, Kubernetes for orchestration, Istio for service mesh, Azure Blob for document storage.

The architecture: five services, each with a single responsibility, communicating through a message broker.

---

## The Five Services

The system decomposed into five bounded contexts:

**Identity Service** — Who is this person? What are they allowed to do? Authentication, authorization, invitation management, organization hierarchy. Owns `identity_db`.

**Document Service** — Where is this document? What does it contain? Upload handling, OCR processing, layout extraction, version management, chunk indexing. Owns `document_db`.

**AI Service** — What does the AI think about this document? Orchestrates the extraction pipeline, manages prompt templates, handles streaming responses, coordinates with the validation worker. No database — it reads from document_db and writes findings to admin_db.

**Admin Service** — What happened, and who approved it? Audit trail, electronic signatures, workflow state, compliance reporting. Owns `admin_db`. This is the regulatory core.

**Analytics Service** — How is the system performing? Review cycle times, finding rates, model accuracy metrics, cost tracking. Owns `analytics_db`.

The message broker sits between the AI service and the document service. It's the boundary between "a document was processed" and "the AI started working on it." That boundary matters for error handling, retry logic, and compliance.

## Why 7 Days?

Seven isn't arbitrary. It's the number we got when we mapped where the 28 days actually went.

- **Days 1–3**: Waiting for the document to be assigned, the reviewer to open it, the initial read-through to happen.
- **Days 4–10**: Active review. This is the irreducible human work — the expert reading, the judgment calls, the domain knowledge applied to the document.
- **Days 11–18**: Back-and-forth between reviewers. Reconciling different findings, discussing edge cases, getting to consensus.
- **Days 19–28**: Report writing, sign-off, archival.

The AI can compress days 1–3 to hours: the moment a document is uploaded, the pipeline starts. The AI can compress days 11–18: if the model surfaces specific findings with specific page references, reviewers are reconciling "do we agree with finding 7?" instead of "did we read the same thing?"

Days 4–10 are the irreducible human core. We don't touch that. We give reviewers better material to work with.

Seven days is what remains after you compress everything except the irreducible human judgment.

## The Compliance Constraint

Every technical decision was made under a constraint: "would this survive an FDA audit?"

That constraint eliminated options quickly.

Can we use streaming responses from the AI? Yes, but we have to log every token that was streamed, including the ones the user never saw. The audit trail is total.

Can we use a managed vector database? Yes, but the database has to be in an approved Azure region, with approved backup policies, with an approved data processing agreement. The vendor has to provide a system qualification package.

Can we let reviewers edit AI findings directly? Yes, but every edit has to be attributed to a specific user, timestamped, and signed. The original AI finding has to be preserved. The audit trail is append-only.

Can we use GPT-4o? Yes, but the Azure OpenAI deployment has to be in a region that supports zero data retention. The model can't learn from our data.

Compliance isn't a feature. It's the shape of the container that everything else fits inside.
